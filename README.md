# 1. Project's Architecture

The data flux follows a simplified "Medallion" architecture:

1. **Source - Bronze:** Raw data in .CSV format (~60k rows x 80 columns - 4M data values)
2. **Processing - Silver - (Python):** Data cleaning, type annotation and data transformation using `Pandas`.
3. **Intermediate Storage - Silver - (Parquet):** Written in tabular format for the most I/O efficiency.
4. **Final Storage - Silver - (Postgres):** Final storage on PostgreSQL.
5. **Treatment of the transformed data - GOLD - (Python):** Appliance of the business rules to the transformed data.
6. **Data Query and visualization - GOLD - (Metabase):** Data analysis via queries.

# 2. Project's Documentation 

All of the dependencies are managed by the **uv** virtual environment and the `pyproject.toml` file.

All of the project's elements are created in docker containers and integrated via `docker-compose`.

## Project's Infrastructure - Docker containers' creation (via Docker-compose)
> **Image 1: Service Integration**
![Services integration](images/Project_Services_integration.png)

This project has 4 services defined in the `docker-compose.yaml` file. They are:

### 1. app:
Defines the Container `Medallion_pipeline`, where the project's Pipeline will run on. It contains the project's `.py` scripts. 

This service depends only on the `postgres-db` service.

At the service's generation step, 4 important actions occur:
- The `main.py` script is executed through the **uv** environment.
- 3 Named volumes are created:
    1. `raw_parquet` volume to persist data from the Bronze Layer. Post extraction (raw-data).
    2. `transformed_silver` volume to persist data from the SIlver Layer. Post data-transformation.
    3. `gx_docs` volume mounted to the local directory to persist data generated by the __great_expectations__ tool.
> **Image 2: app service**
![docker-compose first service](images/docker-compose_service1.png)
 
### 2. postgres-db:
Defines the Container `postgres_db`, where an image of the database __PostgreSQL__ will be created. All of the interaction to the PostgreSQL database (table's creation, queries, etc), will be performed inside of this container.

The `postgres-db` service is connected to the `app` service through a network called __private_network__ which is defined in the `docker-compose` file itself.

At the moment of the service's generation, 3 important actions occur:
- Environment variables are used to supply the necessary information in order to create the PostgreSQL database in the container.
- The named volume `data_sql` is created to persist the data from the container.
- The `init.sql` file is executed to create the project's tables, alongside with the respective data dictionnary. 
> **Image 3: postgres-db service**
![docker-compose second service](images/docker-compose_service2.png)

### 3. metabase:
Defines the `metabase` Container, where the image of the BI tool __Metabase__ will be created.

This container will allow, through a __localhost__ connection, the access of a User Interface (UI) for the visualization of the projects' Dashboard and the execution of other queries that might be of the user's interest.

The `metabase` service connects to the `postgres-db` service through a network called __front_network__ defined in the `docker-compose` file itself.

At the moment of the service's generation, 2 important actions occur:
- The Container path where the files are saved is mapped to the local path `./frontend/metabase_data` so the files can be persisted.
- The port `3000` is exposed to be accessed via localhost.
> **Image 4: metabase service**
![docker-compose third service](images/docker-compose_service3.png)

### 4. web-service:
Defines the `static_web` Container, where an image of the web-server __nginx__ will be created.

This service's Container is connected to the same volume `gx_docs`, generated by the `app` service, to serve the validation report of the data from the Bronze Layer via web, statically.

At the moment of the service's generation, 2 important actions occur:
- The `.html` file present at the local path `gx_docs/gx/uncommitted/data_docs/local_site` is mounted to the Container's path `/usr/share/nginx/html` to be served via web.
- __nginx__ Container's port `80` is mapped to the local port `8080`, where it's exposed to be accessed via localhost.
> **Image 5: web-service service**
![docker-compose fourth service](images/docker-compose_service4.png)

Finally, the project's pipeline was divided into modular steps to guarantee scalability.

These steps are: 

### STEP A: Ingestion (Python)
The data used for the project comes from the OWID - Our World in Data - Organization, that disponibilizes data from the emissions of CO2 gases from various sources worldwide, since the year when the measurement started for each country. 

The raw data was extracted from their github: `https://github.com/owid/co2-data`.

And they were consequently stored at the Container's path `/app/data/raw` and mapped to named volume `raw_parquet` to be persisted as explained above.

### STEP A.1: Data Validation (Great-expectations)
Before following to the Silver layer, the data is validated using the Great Expectations tool.

The data is then persisted to after be served using the Container `web-service`.
> **Image 6: web-service - Great Expectations - Data Docs**
![Great expectations](images/gx_capture.png)

### STEP B: Transformation (Python)
Data is then loaded from the Container's path `/app/data/raw`.

A series of transformations is performed on this data, guaranteeing that the Silver layer is immutable and high-performant.

The data is then saved in `.parquet` format in the Container's path `app/data/silver` and mapped to named volume `transformed_silver` to be persisted.

### STEP C: Load to PostgreSQL (SQLAlchemy)
To the data generated in the Silver Layer the business' rules are applied.

Then the pipeline is connected to the `postgres_db` Container and pushes the tables to the __PostgreSQL__ database.

### STEP D: Analysis and visualization (metabase)
The `metabase` Container is connected directly to the `postgres_db` Container allowing to access the data in an interactive manner.

Through the PostgreSQL's tables, 7 business queries are executed and a dashboard is created.

![Dashboard example](images/dashboard_capture.png)

# 3. Dependencies and Execution Instructions
To run the project you will need to have installed **Docker Engine** (Linux) or **Docker Desktop** (Mac/Windows).

Thanks to Docker, this project is totally containerized and does not require further dependencies' installation.

Before executing the project you will first have to clone the project in your machine:
- Use `git@github.com:GersonCJ/ETL_Medallion_USP_Project.git` via SSH.
- Use `https://github.com/GersonCJ/ETL_Medallion_USP_Project.git` via HTTPS.

After the project has been cloned, make sure to create a `.env` file in the project root folder following the example on `.env.example`. - Or just run on the terminal: `cp .env.example .env`

Optionally, edit the environment variables with your own credentials. If the credentials are not altered, the standard values already allow to the full execution of the project.

To reproduce all of the environment and run the project, just open the terminal and execute the following command:
`docker compose up -d --build`

This command will:
- Build the image for the docker application;
- Start the containers (Postgres, Metabase, Web and App);
- Execute the ETL pipeline;
- Execute the Great Expectations' Validations;
- Generate the HTML documentation of data quality; 
- Disponibilize the BI Dashboard.

After execution, do access:

Great Expectations Data Docs:
http://localhost:8080

Metabase (BI):
http://localhost:3000
- After entering in the Metabase's UI, search for the Dashboard at: `COLLECTIONS -> Shared Collections -> Project Dashboards -> Co2 Project Analysis`

- After stop using the project, make sure to execute in the terminal:
`docker compose down`

This command will:
- Remove the created containers.
